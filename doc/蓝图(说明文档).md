
## 1. **《AI智能知识库助手》**

### 1.0 目标

做企业级的内部知识库助手

### 1.1 功能

- 上传解析 PDF / Word
- 文档内容检索：跟文档聊天、总结、提取重点
- 文档对话流式输出，流式输出前端要有打字机效果
- 保留历史对话

### 1.2 技术栈

**技术栈：`基于 Vue 3 + TS + Vite 做前端，Express 做 Node 后端，要对接 AI - Embedding API 实现 RAG 智能知识库`**

### 1.3 项目约束（核心原则）

1. **业务目标约束**
   - 以“可用、稳定、可演示”为第一目标，优先跑通核心流程。
   - 不追求过度复杂 UI，不实现非必要的花哨功能。

2. **技术选型约束**
   - 全程使用轻量级依赖，不引入重型框架、中间件、数据库。
   - 前后端严格分离，接口标准化、可扩展、易联调。

3. **代码规范约束**
   - 代码结构清晰，按“路由 - 服务 - 工具 - 类型”分层。
   - 所有接口、函数必须使用 TypeScript 类型定义。
   - 核心逻辑可复用、可替换、便于后续升级扩展。

4. **开发优先级约束**
   - 第一优先级：上传解析 → 分块 Embedding → 检索对话 → 流式输出。
   - 第二优先级：历史记录、页面整合、交互优化。
   - 第三优先级：高级功能（多文件、分页、优化）。

一句话总结：**先跑通核心流程、代码结构清晰。**

## 2. 开发每个模块阶段

### 2.1 前端细分小任务（Vue3 + TS + Vite）

#### 任务1：前端项目基础搭建

帮我基于 Vue3 + TS + Vite 初始化前端项目：

- 使用 `npm create vite@latest` 创建项目，选择 Vue + TypeScript 模板；
- 清理默认无用代码（如 HelloWorld 组件、默认样式）；
- 配置基础的 TS 类型校验（如 `tsconfig.json` 补充路径别名、严格模式）；
- 配置 Axios 作为请求库（封装请求拦截器，统一处理请求头、跨域、错误提示）；
- 安装 Pinia（`npm install pinia @pinia/nuxt`），用于内存级存储上传的文本内容，保证页面切换数据不丢；
- 搭建基础路由结构（至少包含「文档上传页」和「智能对话页」）；
- 代码结构按「views- components- utils- api」分层，保留可扩展空间。

#### 任务2：文件上传组件开发

帮我开发独立的 `FileUpload.vue` 组件：

- 基于 Vue3 Composition API + TS 编写，支持选择/拖拽上传 PDF/Word 文件；
- 前端校验文件类型（仅允许 .pdf/.docx）、文件大小（≤10MB），校验失败给出清晰的文字提示；
- 上传时显示进度条，上传成功/失败显示对应状态图标；
- 调用后端上传解析接口（封装到 `api/file.ts` 中），解析返回的文本内容并通过 `emit` 传递给父组件；
- 组件样式简洁（无需复杂UI），仅保留核心交互；
- 代码中添加 TS 类型定义（如文件上传结果类型、错误类型）。

#### 任务3：文本预览组件开发

帮我开发 `TextPreview.vue` 组件：

- 接收父组件传递的解析后文本内容，以只读文本域/滚动容器展示；
- 支持文本长度统计、一键复制功能；
- 空状态（无文本时）显示“请先上传并解析文档”提示；
- 组件适配基础的响应式（宽度自适应）；
- 使用 TS 定义 props 类型（如 `text: string`），确保类型安全。

#### 任务4：对话输入组件开发

帮我开发 `ChatInput.vue` 组件：

- 包含单行输入框（支持回车提交、禁止空内容提交）、发送按钮；
- 输入框聚焦/失焦样式区分，发送按钮在无内容时置灰；
- 点击发送按钮/回车时，通过 `emit` 传递用户输入的问题；
- 使用 TS 定义 emit 事件类型（如 `submit: (question: string) => void`）；
- 输入框绑定值使用 `ref<string>` 类型，保证类型规范。

#### 任务5：对话列表与打字机效果组件

帮我开发 `ChatList.vue` 组件：

- 接收对话列表数据（TS 类型定义：`ChatItem = { id: string; role: 'user' | 'assistant'; content: string }`）；
- 渲染用户/AI 消息气泡（样式区分角色）；
- 接收 AI 流式回答的实时内容，实现**打字机效果**：逐字追加到 AI 消息内容中，回答过程中禁用重复发送；
- 支持空状态（无对话时）显示“请上传文档后提问”提示；
- 代码中封装打字机效果的工具函数（基于 `setTimeout`/`requestAnimationFrame`），保证效果流畅无卡顿。

#### 任务6：流式请求工具函数封装

帮我在 `utils/request.ts` 中封装流式请求函数：

- 基于 Fetch API（而非 Axios，更适配流式响应）编写，接收请求 URL、请求参数（用户问题）；
- 逐行解析流式响应数据（JSON 行格式：每行独立 JSON 字符串，以 `\n` 分隔，示例：`{"chunk":"你好","finish":false}\n{"chunk":"！","finish":true}`）；
- 通过回调函数 `onChunk` 传递每段文本、`onFinish` 标记结束；
- 处理流式请求的异常（如网络中断、后端报错），异常时触发 `onError` 回调；
- 函数添加 TS 类型定义（如流式响应类型、回调函数类型），确保类型安全。

#### 任务7：历史对话列表组件开发

帮我开发 `HistoryChatList.vue` 组件：

- 调用后端历史对话查询接口，渲染历史对话列表（显示问题摘要、创建时间）；
- 点击列表项，将对应问题和回答回显到「对话列表组件」中；
- 支持清空历史对话（调用后端清空接口）；
- 组件处理加载状态、空状态；
- 使用 TS 定义历史对话数据类型；
- 代码中预留后续分页加载的扩展空间。

#### 任务8：页面整合与路由跳转

帮我将上述组件整合到对应页面：

- 「文档上传页」整合 `FileUpload.vue` + `TextPreview.vue`；
- 「智能对话页」整合 `ChatInput.vue` + `ChatList.vue` + `HistoryChatList.vue`；
- 配置路由跳转（上传文档后可一键跳转到对话页）；
- 在 `App.vue` 中添加基础的导航栏（仅包含“上传文档”“智能对话”两个跳转按钮）；
- 确保页面切换时数据不丢失（如上传的文本内容通过 Pinia 简单存储，仅内存级，先跑通流程）。

### 总结

1. 前端拆分原则：每个任务只做一个核心功能（如“仅开发文件上传组件”“仅封装流式请求函数”），降低单任务复杂度；
2. 技术要求：所有组件/函数均使用 Vue3 Composition API + TS 编写，添加完整的类型定义，工具函数/API 调用抽离到独立文件，保证代码结构清晰；
3. 核心优先级：先实现“上传→解析→对话→流式输出”的核心交互，历史对话等功能后置，优先跑通端到端流程。

### 2.2 后端细分小任务（Express + Node.js）

#### 任务1：后端项目基础搭建

帮我基于 Express + TypeScript 初始化后端项目：

- 使用 `npm init` 初始化，安装核心依赖（express、typescript、@types/express、ts-node、cors）；
- 配置 tsconfig.json（启用严格模式、指定输出目录）；
- 搭建基础项目结构（src/routes、src/services、src/utils、src/types）；
- 实现基础的 Express 服务启动逻辑，添加跨域中间件（cors）、请求体解析中间件（express.json()、express.urlencoded()）；
- 服务启动后监听 3000 端口，返回健康检查接口（GET /api/health），返回 { code: 200, msg: 'server is running' }；
- 代码中预留后续中间件扩展空间。

#### 任务2：PDF/Word 上传解析接口开发

帮我在 src/routes 下创建 file.route.ts，实现文件上传解析接口（POST /api/file/upload）：

- 使用 `multer` 处理 multipart/form-data 格式文件上传，限制单文件≤10MB，仅允许 .pdf/.docx 类型；
- PDF 解析使用 `pdf-parse`、Word 解析使用 `mammoth`，封装到 src/services/file.service.ts 中；
- 接口返回结构化数据：{ success: boolean, data: { text: string }, error: string }；
- 包含完整错误处理（文件类型错误、大小超限、解析失败、空文件、multer 上传异常等）；
- 使用 TS 定义请求/响应类型（如 FileUploadResponse、FileParseError），保证类型安全。

#### 任务3：文本分块与 Embedding 生成逻辑开发

帮我在 src/services/embedding.service.ts 中实现核心逻辑：

- 封装文本分块函数：按 500 字切分、上下文重叠 100 字，保留段落完整性（不截断完整句子），TS 定义分块结果类型（TextChunk = { id: string; content: string }）；
- 封装豆包 Embedding API 调用函数：添加 API 密钥配置（从环境变量读取）、超时（10s）、重试（2次）逻辑，TS 定义 Embedding 请求/响应类型；
- 实现“文本→分块→生成 Embedding→内存存储”完整流程：内存使用数组存储 { chunkId: string; content: string; embedding: number[] }，封装新增/查询内存数据的工具函数；
- 暴露接口供后续调用，预留替换为数据库存储的扩展空间。

#### 任务4：相似度计算工具函数开发

帮我在 src/utils/similarity.ts 中实现余弦相似度计算逻辑：

- 编写通用的余弦相似度计算函数，接收两个 embedding 数组（number[]），返回 0-1 之间的相似度值；
- 处理边界情况（embedding 长度不一致、空数组），添加参数校验和错误捕获；
- 使用 TS 定义函数入参/返回值类型；
- 编写简单测试用例验证计算正确性（如固定 embedding 验证相似度结果）。

#### 任务5：文档对话核心接口（非流式）开发

帮我在 src/routes/chat.route.ts 实现非流式对话接口（POST /api/chat/normal）：

- 接收请求参数 { question: string }，TS 定义请求体类型；
- 调用 embedding 服务生成问题的 embedding；
- 计算问题 embedding 与内存中所有文本块 embedding 的相似度，筛选 Top5 高相似度文本块；
- 拼接 Prompt（基于文档内容回答问题），调用豆包对话 API 获取完整回答；
- 返回结构化响应 { success: boolean, data: { answer: string }, error: string }；
- 包含 API 调用、相似度计算、Prompt 拼接的完整错误处理。

#### 任务6：文档对话流式接口开发

帮我在 src/routes/chat.route.ts 扩展流式对话接口（POST /api/chat/stream）：

- 改造豆包对话 API 调用方式为流式请求（设置 stream: true）；
- 通过 Express 的 response.write 逐行返回流式数据，数据格式为 JSON 行（每行独立 JSON 字符串，以 `\n` 分隔，示例：`{"chunk":"你好","finish":false}\n{"chunk":"！","finish":true}`）；
- 设置正确的响应头（Content-Type: text/event-stream, Transfer-Encoding: chunked, Cache-Control: no-cache）；
- 处理流式响应异常（如客户端断开连接、API 流式返回中断），确保资源正常释放；
- 使用 TS 定义流式响应数据类型，封装流式响应工具函数保证复用性。

#### 任务7：历史对话管理接口开发

帮我在 src/routes/history.route.ts 实现历史对话 CRUD 接口（内存存储）：

- 实现新增对话接口（POST /api/history）：接收 { question: string, answer: string }，生成唯一 id 和创建时间，存入内存数组；
- 实现查询所有对话接口（GET /api/history）：返回所有历史对话列表，TS 定义历史对话类型（HistoryItem = { id: string; question: string; answer: string; createTime: string }）；
- 实现清空对话接口（DELETE /api/history）：清空内存中的历史对话数组；
- 所有接口添加参数校验、错误处理；
- 预留后续分页查询、按 id 查询的扩展空间。

#### 任务8：全局异常处理与日志中间件开发

帮我在 src/middlewares 中实现全局中间件：

- 全局异常处理中间件：捕获所有路由/服务中的未处理异常，统一返回 { success: false, error: string }，避免服务崩溃；
- 简单日志中间件：记录每个请求的方法、路径、状态码、响应时间，输出到控制台（无需日志文件，先跑通流程）；
- 将中间件注册到 Express 实例中，保证所有请求经过日志和异常处理。

#### 任务9：环境变量与配置管理

帮我实现后端配置管理：

- 使用 `dotenv` 加载环境变量（如 PORT、DOUBAO_API_KEY、DOUBAO_API_BASE_URL）；
- 在 src/config/index.ts 中封装配置读取函数，统一管理所有配置项，添加配置项缺失的校验；
- TS 定义配置类型，确保配置读取的类型安全，避免硬编码敏感信息（如 API 密钥）；
- AI 任务中调用豆包 API 时，统一从 `src/config/index.ts` 读取 API 密钥/基础 URL，避免重复配置。

### 总结

1. 后端拆分原则：每个任务聚焦单一功能（如“仅开发上传解析接口”“仅实现相似度计算”），降低单任务复杂度，便于分步调试；
2. 技术要求：全程使用 Express + TypeScript 开发，所有接口/函数添加完整 TS 类型定义，按“路由-服务-工具函数”分层，核心逻辑抽离到 service 层保证复用；
3. 核心优先级：先实现“上传解析→分块 embedding→非流式对话”核心链路，再扩展流式接口、历史对话、中间件，优先跑通端到端流程。

### 2.3 AI 细分小任务（对接豆包 API 实现 RAG 智能知识库）

#### 核心目标

聚焦 RAG 核心逻辑，为前后端提供“文本向量化→相似度检索→Prompt 拼接→LLM 调用”能力，轻量依赖，预留扩展空间。

#### 任务1：文本分块 + 豆包 Embedding 生成（后端）

- 分块规则：500 字/块，上下文重叠 100 字，保留段落/句子完整性；
- API 封装：调用豆包 Embedding API，包含 10s 超时、2 次重试、完整错误捕获；
- 存储：结果以 `{text: string, embedding: number[]}` 存入内存数组；
- 逻辑：实现“新增文本块→生成 embedding→追加到内存”闭环，预留数据库扩展接口。

#### 任务2：用户问题检索 + Prompt 拼接（后端 RAG 核心）

- 核心逻辑：接收用户提问 → 生成提问 embedding → 计算余弦相似度 → 筛选 Top5 相关文本块；
- Prompt 拼接：`基于以下文档内容回答问题：{文本块拼接} 问题：{用户提问}`；
- 输出：结构化响应 `{ success: boolean, chunks: string[], error: string }`；
- 边界处理：无文档/无相关内容返回明确提示，不崩溃。

#### 任务3：豆包对话 API 调用（非流式/流式）（后端）

- 非流式：调用豆包对话 API，传入拼接后的 Prompt，返回完整回答；
- 流式：改造为流式调用，逐行返回 `{chunk: string, finish: boolean}`；
- 输出：非流式返回 `{ success: boolean, answer: string, error: string }`，流式通过 `response.write` 推流；
- 异常处理：覆盖 API 密钥缺失、超时、中断等场景，封装工具函数保证复用。

### 总结

1. AI 拆分原则：聚焦 RAG 核心逻辑，不重复前后端已定义的功能，仅实现“文本向量化→检索→Prompt→LLM 调用”核心能力；
2. 技术要求：与后端分层结构对齐，使用 TS 定义类型，封装可复用工具函数，保证代码结构清晰；
3. 核心优先级：先实现非流式 RAG 闭环，再扩展流式调用，优先跑通端到端问答流程。
